# 优化需求分析

## 一、文档背景

当前已经上线并在使用的博客系统为 Halo，但 Halo 的问题在于：

- 自定义能力有限
- 深度功能改造受限
- 很难完全按自己的业务思路扩展

因此，后续计划基于当前这套前后端分离博客项目，逐步补齐后台文章编辑与管理能力，在功能成熟后替换现有 Halo 博客系统。

本阶段目标不是直接编写代码，而是先完成：

- 功能调研
- 可行性分析
- 实现路径梳理
- 风险识别
- 后续优化方向规划

本次分析聚焦于：

## 后台文章编辑与管理模块优化

---

## 二、当前项目文章模块现状概览

从现有项目功能表现来看，当前文章管理已具备以下基础能力：

- 后台写文章
- 编辑文章
- 删除文章
- 预览文章
- Markdown 编辑正文
- 封面上传
- 正文图片上传
- 分类管理
- 标签管理
- 文章前台展示

但与 Halo 相比，当前系统仍然存在明显短板：

1. 保存即发布，缺少草稿状态
2. 不支持多格式文档导入
3. 不支持文章导出
4. 编辑器图片粘贴体验较弱
5. 目录导航能力需要增强

因此，当前项目距离“可替代 Halo 的自研博客系统”仍有明显差距，但整体具备继续演进的基础。

---

## 三、总体结论

### 结论概述

你提出的这 5 项需求，整体上：

## 全部可做

但实现复杂度并不相同，建议分阶段推进，而不是一次性并行开发。

### 推荐优先级

建议优先顺序如下：

1. 草稿与发布分离
2. 编辑器快捷粘贴图片优化
3. 文章结构目录导航
4. 多格式文档导入
5. 文章内容导出

排序原因：

- 前 3 项更贴近当前项目已有结构，改造收益高、依赖少
- 后 2 项涉及格式转换、文件解析、导出渲染，复杂度更高

---

# 四、需求逐项分析

## 需求 1：草稿与发布分离功能

### 1. 需求描述

当前项目中，文章一旦保存，就会直接同步影响前台展示。

期望改造成：

- 保存：仅保存为草稿，前端不可见
- 发布：文章正式发布，对外可见

该逻辑参考 Halo 的文章管理方式。

### 2. 为什么必须做

这是博客后台编辑能力中最基础也最核心的能力之一。

如果没有草稿机制，会产生以下问题：

- 半成品文章误上线
- 修改中的内容直接暴露到前台
- 无法积累待完善的文章素材
- 内容运营流程不完整

如果想真正替代 Halo，这个功能属于基础门槛。

### 3. 实现思路

建议从以下几个层面实现：

#### 数据层
为文章增加状态字段，例如：

- 草稿
- 已发布

后续还可以扩展：

- 已下线
- 定时发布

#### 后端层
后端新增 / 编辑文章接口支持状态参数。

前台查询文章时统一只查询已发布文章。

后台文章列表则应支持查看全部文章，并区分状态。

#### 前端后台层
文章编辑页增加两个动作：

- 保存草稿
- 发布文章

文章列表页增加“状态列”以及后续可考虑增加状态筛选。

### 4. 可行性结论

## 可行性高，建议优先级 P0

因为当前系统已经有文章主体、正文、分类、标签等完整结构，增加的是“状态控制层”，不是重做文章系统。

### 5. 风险与注意点

- 历史文章需要设为默认已发布
- 前台所有文章查询链路都必须统一过滤草稿
- 已发布文章再次编辑后的状态规则需要提前设计清楚
- 后续如果还要做定时发布，状态设计最好预留扩展空间

---

## 需求 2：多格式文档导入

### 1. 需求描述

支持导入 Word、MD 等格式文件，自动解析内容，并适配当前系统的 Markdown 语法。

要求尽量保留：

- 标题层级
- 段落结构
- 排版逻辑
- 列表结构
- 基础格式

最终一键转为博客文章内容。

### 2. 这个功能的价值

这个功能的意义非常大，尤其适合你未来内容迁移和高效生产：

- 可以导入已有 Word 文档
- 可以复用已有 Markdown 内容
- 可以减少人工复制粘贴和重新排版成本
- 可以加快旧平台到新平台的内容迁移

### 3. 实现思路

建议拆阶段设计：

#### 第一阶段：先支持 MD 文件导入

这是最容易落地的版本。

原因：

- 当前系统正文本身就是 Markdown 体系
- MD 文件本质就是文本读取
- 转换成本最低
- 兼容性最高

实现方式可以是：

- 后台上传 `.md` 文件
- 服务端读取文本内容
- 自动填充到文章编辑器正文区域
- 可选解析标题为文章标题

#### 第二阶段：支持 Word 导入

Word 导入建议优先支持 `.docx`，不优先考虑老旧 `.doc`。

实现路径通常为：

- 读取 Word 文档结构
- 提取段落、标题、列表、图片
- 转换为 Markdown 或 HTML
- 再适配到当前文章编辑器中

#### 第三阶段：格式兼容优化

后续再逐步优化：

- 表格转换
- 图片抽取与上传
- 引用块兼容
- 代码块兼容
- 特殊格式兜底处理

### 4. 可行性结论

## 可行，但复杂度中高，建议分阶段实现

### 5. 风险与注意点

- Word 格式非常复杂，不可能 100% 无损还原
- 不同 Word 文档结构差异大，兼容性成本较高
- 图片、表格、嵌套列表的还原难度较高
- 导入后仍可能需要人工微调
- 如果直接把 Word 转 Markdown，部分样式信息会丢失

### 6. 产品建议

建议对外明确定位为：

- 辅助导入
- 结构保留优先
- 不承诺绝对还原

---

## 需求 3：文章内容导出功能

### 1. 需求描述

实现文章导出，支持：

- MD
- PDF

方便内容备份、迁移和离线查看。

### 2. 这个功能的价值

导出功能的价值主要体现在：

- 内容安全备份
- 平台迁移方便
- 离线归档方便
- 便于二次分发和整理

### 3. 实现思路

#### MD 导出

这是最简单、最建议先做的部分。

因为当前系统正文本身就是 Markdown 存储，所以导出 MD 本质上几乎是直接导出正文内容。

可附加导出信息包括：

- 标题
- 摘要
- 标签
- 分类
- 封面链接
- 正文 Markdown

#### PDF 导出

PDF 导出复杂度更高，一般需要：

- 先把文章整理为标准 HTML
- 再将 HTML 渲染为 PDF

也可以考虑前端导出或服务端导出两种方式。

### 4. 可行性结论

## 可行

其中：

- MD 导出：可行性高，复杂度低
- PDF 导出：可行性中等，复杂度中高

### 5. 风险与注意点

- PDF 样式一致性需要单独设计
- 不同系统环境生成 PDF 的结果可能不同
- 包含图片、长代码块、分页时，需要额外处理
- 如果文章很长，PDF 的分页与目录体验需要优化

### 6. 产品建议

建议先做：

1. 单篇文章导出 MD
2. 单篇文章导出 PDF
3. 后续再扩展批量导出

---

## 需求 4：编辑器快捷粘贴图片优化

### 1. 需求描述

希望复刻 Halo 的便捷体验，支持：

- 截图复制后直接粘贴
- 剪贴板图片直接粘贴
- 自动上传图片
- 自动插入文章内容

### 2. 这个功能的价值

这个功能虽然看起来是体验优化，但实际价值非常高：

- 极大减少配图操作步骤
- 提高写作流畅度
- 更适合高频截图写作
- 对教程类、技术类文章价值特别大

### 3. 实现思路

核心思路通常是：

- 监听编辑器粘贴事件
- 从剪贴板中识别图片对象
- 自动调用上传接口
- 上传成功后把图片 Markdown 链接插入当前光标位置

如果当前编辑器本身已有上传接口能力，那么这个功能很可能属于在现有基础上增强交互层。

### 4. 可行性结论

## 可行性高，建议优先级 P1

它相较于文档导入、PDF 导出要简单很多，但用户体验收益很大。

### 5. 风险与注意点

- 不同浏览器对剪贴板支持存在差异
- 大图片粘贴时要考虑压缩与上传耗时
- 需要处理上传失败时的提示与回滚逻辑
- 需要考虑与现有编辑器快捷键、默认粘贴行为是否冲突

### 6. 产品建议

建议与图片上传模块一并优化：

- 粘贴即上传
- 上传中有状态提示
- 上传失败可重试
- 自动返回图片链接并插入正文

---

## 需求 5：文章结构目录导航

### 1. 需求描述

在后台编辑页、前端文章详情页右侧增加标题目录栏。

要求：

- 自动识别全文标题层级
- 生成目录结构
- 支持点击快速定位跳转
- 优化长文编辑与阅读体验

### 2. 这个功能的价值

这项能力对长文非常重要：

- 编辑时便于快速定位章节
- 阅读时便于理解全文结构
- 有助于提升内容专业感
- 对教程、笔记、系列文章非常友好

### 3. 实现思路

#### 前端文章详情页

前台通常是最容易实现的，因为文章展示前已经是 HTML 结构，可以直接识别标题节点：

- H1
- H2
- H3
- H4

再根据这些标题生成目录。

#### 后台编辑页

后台编辑器目录实现稍复杂一些，因为编辑中的内容是动态变化的。

一般思路是：

- 实时解析当前 Markdown 内容
- 提取标题层级
- 生成实时目录树
- 点击目录后定位编辑器中的对应位置

### 4. 可行性结论

## 可行

其中：

- 前台目录导航：可行性高，复杂度较低
- 后台编辑目录导航：可行性高，复杂度中等

### 5. 风险与注意点

- 编辑器内部光标定位能力要确认是否易于接入
- 标题重名时的定位锚点要处理好
- 标题层级过深时目录展示要做好折叠或缩进
- 需要考虑移动端表现

### 6. 产品建议

建议分两步做：

1. 先做前台文章详情页目录导航
2. 再做后台编辑页实时目录导航

---

# 五、整体实施建议

## 第一阶段（优先做）

1. 草稿与发布分离
2. 编辑器快捷粘贴图片优化
3. 前台文章目录导航

原因：

- 与当前系统结构更贴近
- 用户价值高
- 改造风险相对可控

## 第二阶段（中期推进）

4. 后台编辑页目录导航
5. 文章导出 MD
6. 多格式导入先做 MD 导入

## 第三阶段（复杂能力）

7. Word 导入
8. PDF 导出
9. 文档格式兼容优化

---

# 六、关键风险汇总

后续优化时要重点关注以下风险：

1. 文章状态改造可能影响前台全部文章查询链路
2. 文档导入无法做到完全无损，需控制预期
3. PDF 导出样式一致性要求较高
4. 粘贴图片功能存在浏览器兼容差异
5. 目录导航需要处理标题层级、重名、定位精度问题

---

# 七、最终结论

从产品方向上看，这 5 项需求都非常合理，而且都能明显提升当前系统替代 Halo 的可能性。

其中：

- `草稿与发布分离` 是基础能力，必须优先做
- `快捷粘贴图片` 和 `目录导航` 是高收益体验优化
- `多格式导入` 和 `文章导出` 是增强型能力，能形成你这个系统区别于 Halo 的特色优势

整体上，这些需求：

## 值得做、可以做、但应分阶段推进

不建议一次性同时开发，而应遵循“先基础流程、再体验优化、再复杂格式能力”的路线逐步落地。
